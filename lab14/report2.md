## Теоретическая часть

Документные базы данных (NoSQL) хранят данные в виде гибких, полуструктурированных документов (обычно JSON/BSON). В отличие от реляционных СУБД, здесь не требуется фиксированная схема: каждый документ может иметь свой набор полей. Связанные данные часто хранятся вместе (денормализация), что позволяет получать информацию за один запрос без выполнения JOIN.

**MongoDB** – самая популярная документная СУБД. Ключевые понятия:
- Коллекция – аналог таблицы, но без жёсткой схемы.
- Документ – аналог строки, хранится в BSON-формате.
- Вложенные документы и массивы – естественный способ представления иерархических данных (например, корзина заказа).
- Агрегационный пайплайн – последовательность этапов ($match, $group, $lookup и др.) для обработки данных.
- Индексы – аналогичны реляционным, но адаптированы под документную модель.
В работе применялись:
- $lookup – аналог LEFT JOIN для связывания коллекций.
- $unwind – «разворачивание» массива в отдельные документы.
- $group – агрегация с $sum, $avg.
- $addFields, $project, $sort, $limit.
- $map – вычисления внутри массива.
- updateMany с агрегационным пайплайном для атомарного обновления

### Ключевые фрагменты кода

#### 1. Вставка данных с вложенными структурами (пример заказа)
```javascript
db.orders.insertMany([
    {
        _id: 1,
        user_id: 1,                     // ссылка на пользователя
        order_date: new Date("2026-04-15"),
        status: "completed",
        items: [                        // вложенный массив товаров
            { product_id: 1, quantity: 1, price: 75000 },
            { product_id: 2, quantity: 2, price: 1500 }
        ]
    },
    // ... другие заказы
]);
```

#### 2. READ: заказы Alice с суммой (используем $lookup, $unwind, $map)
```javascript
db.orders.aggregate([
    { $lookup: { from: "users", localField: "user_id", foreignField: "_id", as: "user" } },
    { $unwind: "$user" },
    { $match: { "user.email": "alice@example.com" } },
    {
        $addFields: {
            total_amount: {
                $sum: {
                    $map: {
                        input: "$items",
                        as: "item",
                        in: { $multiply: ["$$item.quantity", "$$item.price"] }
                    }
                }
            }
        }
    },
    { $project: { order_id: "$_id", status: 1, total_amount: 1, "user.full_name": 1, _id: 0 } }
]);
```

#### 3. UPDATE: добавление скидки для заказов дороже 80000
```javascript
// Сначала добавляем поле total_amount всем заказам (агрегационное обновление)
db.orders.updateMany(
    {},
    [
        {
            $set: {
                total_amount: {
                    $sum: {
                        $map: {
                            input: "$items",
                            as: "i",
                            in: { $multiply: ["$$i.quantity", "$$i.price"] }
                        }
                    }
                }
            }
        }
    ]
);
// Затем применяем скидку 10%
db.orders.updateMany(
    { total_amount: { $gt: 80000 } },
    { $set: { discount: 10 } }
);
```
#### 4. DELETE: удаление отменённых заказов старше 30 дней
```javascript
const cutoff = new Date();
cutoff.setDate(cutoff.getDate() - 30);
db.orders.deleteMany({
    status: "cancelled",
    order_date: { $lt: cutoff }
});
```
#### 5. Агрегация по категориям (выручка, количество, средняя цена)
```javascript
db.orders.aggregate([
    { $unwind: "$items" },
    {
        $lookup: {
            from: "products",
            localField: "items.product_id",
            foreignField: "_id",
            as: "product"
        }
    },
    { $unwind: "$product" },
    {
        $group: {
            _id: "$product.category",
            total_quantity: { $sum: "$items.quantity" },
            total_revenue: { $sum: { $multiply: ["$items.quantity", "$items.price"] } },
            avg_price: { $avg: "$items.price" }
        }
    },
    { $sort: { total_revenue: -1 } },
    {
        $project: {
            category: "$_id",
            total_quantity: 1,
            total_revenue: 1,
            avg_price: { $round: ["$avg_price", 2] },
            _id: 0
        }
    }
]);
```
#### 6. Топ-3 пользователей по сумме заказов (аналог SQL CTE)
```javascript
db.orders.aggregate([
    { $unwind: "$items" },
    {
        $group: {
            _id: "$user_id",
            total_spent: { $sum: { $multiply: ["$items.quantity", "$items.price"] } }
        }
    },
    { $sort: { total_spent: -1 } },
    { $limit: 3 },
    {
        $lookup: {
            from: "users",
            localField: "_id",
            foreignField: "_id",
            as: "user"
        }
    },
    { $unwind: "$user" },
    {
        $project: {
            full_name: "$user.full_name",
            total_spent: 1,
            _id: 0
        }
    }
]);
```
#### 7. Создание индексов и анализ выполнения
```javascript
db.orders.createIndex({ user_id: 1 });
db.orders.createIndex({ status: 1, order_date: -1 });
db.products.createIndex({ name: "text" });

// Анализ запроса с индексом
db.orders.find({ user_id: 1 }).explain("executionStats");
```

## Ответы на контрольные вопросы
1. **Как в MongoDB представлена связь «заказ-товары» в отличие от PostgreSQL?**  
   В PostgreSQL связь реализована через отдельную таблицу order_items с внешними ключами на orders и products (нормализованная модель). В MongoDB связь представлена как вложенный массив items внутри документа заказа. Каждый элемент массива содержит product_id, quantity и цену на момент заказа. Такой подход позволяет получить полную информацию о заказе одним запросом без JOIN.
2. **В каком случае документная модель удобнее реляционной? Приведите пример из работы.**  
  Документная модель удобнее, когда данные естественно иерархичны и часто извлекаются вместе, а также когда схема часто меняется. В нашей работе хранение товаров заказа как вложенного массива удобнее, чем отдельная таблица: для отображения страницы заказа нужен один запрос к коллекции orders вместо трёх таблиц с JOIN. Также в MongoDB легко добавить новое поле (например, discount) без миграции всей схемы.
3. **Какие операции в MongoDB оказались сложнее/проще по сравнению с SQL?**  
   Проще: вставка сложных вложенных документов (одна операция вместо нескольких `INSERT`). Сложнее: агрегация с `$unwind` и `$lookup` требует большего количества этапов и понимания пайплайна, в то время как SQL-запрос с `JOIN` и `GROUP BY` выглядит компактнее и привычнее.

4. **Что такое `$unwind` и зачем он нужен?**  
 $unwind деструктурирует массив: из одного документа с массивом создаётся множество документов, каждый из которых содержит один элемент массива. Это необходимо перед такими операциями, как $group по полям внутри массива. Например, для подсчёта количества проданных товаров по категориям нужно сначала «развернуть» items, чтобы каждый товар стал отдельным документом, затем выполнить $lookup с продуктами и $group.

## Теоретическая часть
**ETL (Extract, Transform, Load)** — это процесс, используемый для интеграции данных из различных источников в единое хранилище. Он включает три этапа:
- Extract (извлечение) — чтение данных из исходных систем (файлы, базы данных, API).
- Transform (трансформация) — очистка, нормализация, агрегация, обогащение данных.
- Load (загрузка) — запись обработанных данных в целевую систему (обычно базу данных или хранилище).

В данной работе ETL-пайплайн реализован на Python с использованием библиотек pandas, sqlalchemy, matplotlib, seaborn. Источник данных — CSV-файл с записями о продажах интернет-магазина. Целевая система — SQLite (две таблицы: очищенные данные и агрегированные результаты). Дополнительно выполняется визуализация полученных агрегатов.

**Преимущества ETL-подхода:**
- Контроль качества данных до загрузки.
- Возможность сложных преобразований, не поддерживаемых целевой БД.
- Независимость от конкретной СУБД.

**Основные этапы ETL:**
- **Extract** – чтение исходных данных (CSV, API, БД) с обработкой ошибок и проверкой структуры.
- **Transform** – удаление дубликатов, заполнение пропусков, фильтрация аномалий, приведение типов, создание вычисляемых полей.
- **Load** – загрузка очищенных и агрегированных данных в хранилище (в работе — SQLite).

### Ключевые фрагменты кода

**1. Извлечение данных (метод extract)**
```python
def extract(self):
    try:
        self.raw_data = pd.read_csv(self.csv_path)
        logger.info(f"Загружено {len(self.raw_data)} строк, {len(self.raw_data.columns)} колонок")
    except FileNotFoundError:
        logger.error(f"Файл {self.csv_path} не найден")
        raise
    return self.raw_data
```

**2. Очистка и трансформация (метод transform)**
```python
# Удаление дубликатов
df = df.drop_duplicates()

# Заполнение числовых пропусков медианой
for col in ['quantity', 'price_per_unit']:
    df[col] = df[col].fillna(df[col].median())

# Заполнение текстовых пропусков
for col in ['category', 'product_name', 'customer_name']:
    df[col] = df[col].fillna("Unknown")

# Удаление аномалий
df = df[(df['quantity'] > 0) & (df['price_per_unit'] > 0)]

# Преобразование типов
df['order_date'] = pd.to_datetime(df['order_date'], errors='coerce')
df['quantity'] = pd.to_numeric(df['quantity'], errors='coerce')
df['price_per_unit'] = pd.to_numeric(df['price_per_unit'], errors='coerce')

# Добавление колонок
df['total_amount'] = df['quantity'] * df['price_per_unit']
df['month_year'] = df['order_date'].dt.strftime('%Y-%m')
```

**3. Агрегация (метод aggregate)**
```python
self.aggregated_data = df.groupby(['category', 'month_year']).agg({
    'quantity': 'sum',
    'total_amount': 'sum',
    'price_per_unit': 'mean',
    'order_id': 'nunique'
}).rename(columns={
    'quantity': 'total_quantity',
    'total_amount': 'total_revenue',
    'price_per_unit': 'avg_price',
    'order_id': 'order_count'
}).reset_index()
```

**4. Загрузка в SQLite (метод load_to_sqlite)**
```python
engine = create_engine(f'sqlite:///{self.db_path}')
self.cleaned_data.to_sql('sales_cleaned', engine, if_exists='replace', index=False)
self.aggregated_data.to_sql('sales_aggregated', engine, if_exists='replace', index=False)
```

**5. Визуализация (фрагмент)**

category_revenue = self.aggregated_data.groupby('category')['total_revenue'].sum()

2. **В чём разница между ETL и ELT? Какой подход для данной задачи?**  
   ETL выполняет трансформацию перед загрузкой в хранилище, ELT – после загрузки. В работе использован классический ETL, так как данные небольшие и очистка в pandas перед загрузкой в SQLite проще.

3. **Почему в реальных проектах данные не загружают напрямую в базу без очистки?**  
Потому что неочищенные данные вызывают:

- Неверные отчёты и метрики (например, дубликаты завышают выручку, отрицательные суммы искажают средний чек).
- Нарушение целостности БД (NULL в NOT NULL полях, неверные внешние ключи).
- Потерю доверия к данным со стороны бизнес-пользователей.
- Снижение производительности запросов из-за «мусорных» строк и неоптимальных типов.

Очистка – обязательный этап любого промышленного конвейера данных.

- Извлечение дня недели из order_date – для анализа, в какие дни продажи выше.

- Нормализация названий городов – например, «СПб» → «Санкт-Петербург».
- Удаление заказов с аномально высокими ценами (выбросы).
- Добавление колонки is_weekend – для сравнения продаж в будни и выходные.
- Флаги возвратов – если бы данные содержали возвраты, нужно было бы отдельно их обрабатывать.
- Интеграция с курсами валют – если цены указаны в разных валютах.

#### 1. Теоретическая часть
**1.1. Анализ зависимостей (SCA – Software Composition Analysis)**
**SCA** — метод анализа безопасности, который сканирует зависимости проекта (библиотеки, пакеты, фреймворки) на наличие известных уязвимостей (CVE). Инструменты SCA:

- Анализируют манифесты зависимостей (package.json, requirements.txt, pom.xml)
- Сверяют версии с базами данных уязвимостей (NVD, GitHub Advisory Database)
- Выявляют лицензионные ограничения и конфликты

**Основные инструменты SCA для Node.js:**
- npm audit — встроенный в npm
- snyk — расширенный анализ с мониторингом
- OWASP Dependency-Check

**1.2. Динамическое тестирование (DAST – Dynamic Application Security Testing)**
DAST — метод тестирования безопасности работающего приложения ("чёрный ящик"). Инструменты DAST:

- Не требуют доступа к исходному коду
- Отправляют HTTP-запросы с вредоносными payload'ами
- Анализируют ответы на наличие признаков уязвимостей

**OWASP ZAP (Zed Attack Proxy)** — популярный open-source инструмент DAST, поддерживающий:

- Пассивное сканирование (анализ трафика без атак)
- Активное сканирование (отправка специальных запросов)
- Автоматизацию через CI/CD

**1.3. XSS (Cross-Site Scripting)**
XSS — уязвимость, при которой злоумышленник внедряет вредоносный JavaScript-код в веб-страницу.

**Типы XSS:**

Тип-Описание
Reflected-Скрипт передаётся в параметре URL и отражается в ответе
Stored-Скрипт сохраняется на сервере (например, в комментариях)
DOM-based-Скрипт выполняется на клиенте через манипуляции с DOM
**Защита от XSS:**

1.Экранирование вывода (HTML escape)

2.Content Security Policy (CSP)

3.Санитизация входных данных

4.Использование textContent вместо innerHTML

### 3)Ключевые фрагменты кода
**3.1. Исходный (уязвимый) код**
**SQL-инъекция (конкатенация):**
```javascript
// Уязвимо
const search = req.query.q || '';
db.all(`SELECT * FROM comments WHERE comment LIKE '%${search}%'`);

// Уязвимо (сортировка без allow-list)
const sort = req.query.sort || 'created_at DESC';
db.all(`SELECT * FROM comments ORDER BY ${sort}`);
```

**XSS (небезопасная вставка):**
```javascript
// Фронтенд
html += '<div>' + comment.comment + '</div>';
document.getElementById('api-result').innerHTML = html;
```

**Hardcoded секрет:**
```javascript
const API_KEY = 'test-api-key-12345';
```

**SSRF (нет валидации URL):**
```javascript
const url = req.query.url || '';
const response = await axios.get(url);
```
**3.2. Исправленный (безопасный) код**
**Параметризованный запрос:**
```javascript
db.all(`SELECT * FROM comments WHERE comment LIKE ?`, [`%${search}%`]);
```

**Allow-list для сортировки:**
```javascript
const allowedSort = ['created_at DESC', 'created_at ASC', 'username ASC', 'username DESC'];
if (!allowedSort.includes(sortParam)) {
    return res.status(400).json({ error: 'Invalid sort parameter' });
}
```

**Санитизация HTML:**
```javascript
function sanitizeHtml(str) {
    return str.replace(/[&<>]/g, function(m) {
        if (m === '&') return '&amp;';
        if (m === '<') return '&lt;';
        if (m === '>') return '&gt;';
        return m;
    });
}
// При сохранении комментария
username = sanitizeHtml(username || 'Anonymous');
comment = sanitizeHtml(comment || '');
```

**Безопасная вставка на клиенте:**
```javascript
const div = document.createElement('div');
div.textContent = comment.comment;  // вместо innerHTML
container.appendChild(div);
```

## Ответы на контрольные вопросы
1. **Какие уязвимости обнаружены в зависимостях?**  
   Были найдены критические и высокие уязвимости:  
   - axios@0.18.0 — HTTP Response Splitting (CVE-...), Prototype Pollution, SSRF  
   - ejs@2.6.1 — Remote Code Execution  
   - express@4.16.0 — Open Redirect  
   - body-parser@1.18.3 — Amplification  
   - sqlite3@5.0.0 — Arbitrary Code Execution, Path Traversal  

2. **В чем разница между `npm audit` и `snyk test`?**  
   Оба анализируют зависимости, но `npm audit` использует базу GitHub Advisory, ограниченную прямыми зависимостями. Snyk анализирует полное дерево зависимостей, находит более глубокие уязвимости, даёт рекомендации по обновлению и предоставляет дополнительную информацию (лицензии, уровень серьёзности).

3. **Как OWASP ZAP обнаруживает XSS-уязвимости?**  
   В пассивном режиме ZAP анализирует HTTP-трафик и ищет отражение введённых данных без экранирования. В активном режиме отправляет XSS-векторы и проверяет, выполнился ли скрипт (появление alert, изменение DOM).

4. **Почему CSP эффективен против XSS, даже если разработчик забыл экранировать вывод?**  
   CSP запрещает выполнение встроенных скриптов без указания `nonce` или `'unsafe-inline'`. Если злоумышленник внедрит тег `<script>`, браузер проигнорирует его, так как он не содержит разрешённого nonce или не загружен с доверенного источника.

5. **Какие ограничения есть у DAST-инструментов по сравнению с SAST?**  
   DAST тестирует только работающее приложение, не видит уязвимости в неактивных участках кода и зависит от покрытия тестами. SAST анализирует исходный код, находит проблемы на этапе написания, но может давать ложные срабатывания и не учитывать среду выполнения.

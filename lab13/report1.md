## Теоретическая часть
Статический анализ кода (SAST) — это метод поиска уязвимостей без выполнения программы. Bandit — open-source‑инструмент для Python, который находит распространённые проблемы безопасности: SQL‑инъекции, использование `shell=True`, хранение секретов в коде, небезопасные импорты и т.д. В ходе работы изучались:
- Механизмы внедрения SQL‑кода через конкатенацию строк и защита параметризованными запросами.
- Опасность выполнения команд ОС с `shell=True` и альтернатива с передачей аргументов списком.
- Принцип хранения секретов в переменных окружения вместо жёсткого кодирования.
- Настройка Bandit через конфигурационный файл `.bandit` для игнорирования ложных срабатываний и установки порога серьёзности.

**Исправление SQL-инъекции (до/после):**

```python
cursor.execute(query)

# Hardcoded секрет

# Command injection

# Flask запуск

# После (безопасный параметризованный запрос)
# Параметризованный запрос (SQL-инъекция устранена)
user_id = request.args.get('id')
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, (user_id,))

# Секрет из переменной окружения
API_KEY = os.environ.get('API_KEY')
if not API_KEY:
    raise ValueError("API_KEY environment variable not set")

# Command injection с allow-list
ALLOWED_COMMANDS = ['whoami', 'hostname', 'ver', 'ping']
parts = cmd.split()
if parts[0] not in ALLOWED_COMMANDS:
    return jsonify({"error": "Command not allowed"}), 403
result = subprocess.check_output(parts, stderr=subprocess.STDOUT, text=True)

# Безопасный запуск
app.run(debug=False, host='127.0.0.1', port=5000)

```

2. **Почему параметризованные запросы защищают от SQL-инъекций?**  
   Параметризованные запросы отделяют структуру SQL-команды от данных. Драйвер базы данных передаёт значения отдельно, автоматически экранируя спецсимволы, поэтому пользовательский ввод никогда не интерпретируется как SQL-код. Даже если передать 1 OR 1=1, БД будет искать строку, равную этому значению, а не выполнять условие.

3. **В чем разница между `subprocess.run(cmd, shell=True)` и `subprocess.run(cmd.split())` с точки зрения безопасности?**  
   - shell=True — команда выполняется через системную оболочку (cmd.exe или /bin/sh). Оболочка интерпретирует метасимволы (;, |, &&, $), что позволяет злоумышленнику выполнить произвольные команды, например 8.8.8.8; rm -rf /.

- cmd.split() + shell=False (по умолчанию) — команда запускается напрямую, без оболочки. Однако сама по себе эта практика небезопасна, если не валидировать аргументы. Нужен allow-list и избегание прямого вызова пользовательского ввода.

4. **Какие уязвимости статический анализатор (SAST) не может обнаружить? Приведите примеры.**  
- Логические ошибки авторизации (IDOR): пользователь может подставить чужой ID в URL, а сервер не проверит права.

- Проблемы конфигурации: открытый S3-бакет, неправильные CORS-заголовки.

- Уязвимости времени выполнения (race conditions): при одновременных запросах состояние может быть нарушено.

- Ошибки бизнес-логики: возможность дважды потратить баланс, применить купон больше одного раза.

- Уязвимости в зависимостях (SCA): если библиотека содержит CVE, SAST не увидит, пока не проверит код самой библиотеки.
